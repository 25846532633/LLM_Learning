# 一、SingleHead Self-Attention（单头自注意力）答案提纲

## 1. 它想解决什么问题
- embedding 只能给出每个 token 自己的静态表示，不能直接建模 token 和 token 之间的相关性。
- self-attention 的目标，是让每个 token 都能根据当前上下文，动态决定该看谁、看多少。
- “self” 指的是：Q、K、V 都来自同一份输入序列 `x`。
- 一句话本质：**每个 token 用自己的 query 去和所有 token 的 key 匹配，再用得到的权重对所有 value 做加权汇总。**

## 2. Q / K / V 怎么理解
- `Q`：我现在想找什么信息。
- `K`：我这里有什么信息，可不可以被你关注。
- `V`：真正被取出来、被聚合的内容。
- 之所以不直接对 `x` 运算，而要先线性投影，是为了让模型在不同子空间中学习“匹配规则”和“输出内容”。

## 3. Shape 主线
- 输入常见 shape：`[B, T, C]`
- 经过 Q/K/V 线性层后，shape 通常仍是 `[B, T, C]`
- `K.transpose(-2, -1)` 后变成 `[B, C, T]`
- `Q @ K^T` 得到 score：`[B, T, T]`
- `softmax(dim=-1)`：表示每个 token 对所有 token 的关注分布
- `attn @ V` 后输出又回到 `[B, T, C]`
- token 数量 `T` 不变，因为 attention 是“信息混合”，不是“删 token / 加 token”

## 4. 核心公式
```text
Q = XW_Q
K = XW_K
V = XW_V
score = QK^T / sqrt(d_k)
attn = softmax(score)
out = attnV
```

## 5. 为什么要缩放
- 点积维度大时，score 容易数值过大。
- softmax 会变得过尖锐，梯度不稳定。
- 所以要除以 `sqrt(d_k)`，让分布更平稳。

## 6. Mask 的核心理解
- mask 加在 **score** 上，不加在 Q/K/V 上。
- 常见写法是把不该看的位置填成 `-inf`，再做 softmax，让这些位置权重变成 0。
- padding mask：不看补齐位置。
- causal mask：不看未来位置。
- 如果某一整行全被 mask，softmax 可能出现 `NaN`。

## 7. 30 秒口述模板
- 输入 `x` 先映射成 Q/K/V。
- Q 和 K 算相关性，得到每个 token 对其他 token 的打分。
- 打分缩放后做 softmax，变成注意力权重。
- 再用这些权重对 V 做加权求和，得到新的 token 表示。
- 所以 attention 本质上是“按相关性做信息聚合”。

# 二、MultiHead Self-Attention（多头自注意力）答案提纲

## 1. 为什么从单头变多头
- 单头只能在一个表示子空间里做一次注意力。
- 多头让模型可以并行学习不同类型的关系：语义关系、位置关系、长距离依赖等。
- 重点不是单纯“参数变多”，而是 **表示能力更丰富**。

## 2. 多头到底“多”在哪
- 不是把 attention 简单重复多次。
- 而是把 hidden dimension 拆成多个 head，每个 head 都在自己的子空间里做 attention。
- 每个 head 可以理解为“一种不同的观察角度”。

## 3. Shape 主线
- 输入：`[B, T, C]`
- 线性投影后：`[B, T, C]`
- reshape 为：`[B, T, H, D]`，其中 `C = H * D`
- transpose 后常写为：`[B, H, T, D]`
- score：`[B, H, T, T]`
- 每个 head 独立算 attention
- concat 后：`[B, T, C]`
- 再过 `output_proj`，完成不同 head 信息融合

## 4. 核心理解
- 多头 attention 的流程和单头一样，只是变成了“多个单头并行”。
- 更标准的缩放因子是 `sqrt(head_dim)`，因为每个 head 的点积维度是 `head_dim`。
- 如果误写成 `sqrt(hidden_dim)`，理论上缩放过度，会影响注意力分布。

## 5. output_proj 的作用
- 把所有 head 拼接后的结果重新混合。
- 让不同 head 提取到的信息在输出空间里重新整合。
- 没有这一层，多头只是“并排放着”，融合能力不足。

## 6. 多头的收益与代价
- 收益：表达更强、可学到更多关系模式。
- 代价：实现更复杂，计算和显存开销更高。
- 头数不是越多越好：head 太多会让每个 `head_dim` 太小，单个 head 表达能力反而受限。

## 7. Mask 里的易错点
- 多头下 mask 一般要对齐到 `[B, H, T, T]` 或能广播到这个 shape。
- 一行全 `-inf` 仍可能导致 softmax 出 `NaN`。
- 工程里通常会保证至少有一个可见位置，或者对 mask 做更稳健处理。

## 8. 一句话对比 SingleHead
- SingleHead：一个角度看全序列。
- MultiHead：多个角度并行看全序列，再把结果融合起来。

# 三、Transformer Decoder（解码器）答案提纲

## 1. Decoder block 的核心组成
- Masked Self-Attention
- Add & Norm（残差 + LayerNorm）
- FFN / MLP
- 再一次 Add & Norm

## 2. 每一部分的分工
- attention：负责 **token 与 token 之间的信息交互**
- FFN：负责 **每个 token 内部的非线性特征变换**
- residual：保留原信息，帮助深层训练
- LayerNorm：稳定数值分布，提升训练稳定性

## 3. 为什么 decoder 需要 causal mask
- 因为 decoder 是自回归生成。
- 当前 token 只能看自己和过去，不能偷看未来。
- 如果不加 causal mask，训练时模型会看到答案后面的信息，和生成时不一致。

## 4. 为什么训练能并行，推理却不能
- 训练时目标序列已知，可以一次把所有位置都送进去，再用 causal mask 限制可见性。
- 推理时下一个 token 还没生成，只能一步一步往后生成。

## 5. FFN 为什么重要
- attention 主要做 token 间交互，但不擅长复杂的逐 token 非线性变换。
- FFN 相当于在每个位置上单独做更强的特征加工。
- 所以 Transformer 的能力不只来自 attention，也来自 FFN。

## 6. Shape 主线
- block 输入通常是 `[B, T, C]`
- self-attention 前后 shape 不变
- FFN 前后 shape 对外也不变
- 即使 FFN 内部会短暂升维，block 对外输出仍保持 `[B, T, C]`
- 多层堆叠时，shape 一致，便于层层堆叠

## 7. 教学版 decoder 与真实 LLM 的差距
- 常缺：位置编码 / RoPE、KV cache、dropout 细节、weight tying、更多工程优化
- 教学版足够理解数据流，但不等于工业级实现

## 8. 一个 block 和整个 decoder 的区别
- 一个 decoder block：一层结构单元
- 整个 decoder：embedding + 多层 block + 最终输出头
- LLM 通常是很多个 decoder block 堆叠起来

## 9. 一句话口述模板
- Decoder block 先用 masked self-attention 做上下文交互，再用 FFN 做逐 token 特征变换，中间都配合残差和归一化来保证深层训练稳定。

# 四、Grouped Query Attention（GQA / Group Attention）答案提纲

## 1. 为什么会有 GQA
- 标准 MHA 中，每个 query head 都有独立 K/V，推理时 KV cache 很大。
- GQA 的核心目标，是在尽量保留效果的同时，减少 K/V 数量，降低推理成本。
- 它更偏向 **推理阶段优化**，尤其是长上下文生成时的显存和带宽压力。

## 2. MHA / GQA / MQA 的核心差别
- MHA：每个 Q head 都有自己独立的 K/V
- GQA：多个 Q head 分组共享 K/V
- MQA：所有 Q head 共享同一组 K/V
- 本质差别就是：**K/V 共享到什么程度**

## 3. 三者关系
- MHA：表达最强，但开销最大
- MQA：开销最小，但可能损失更多表达能力
- GQA：在两者之间折中
- 当 `num_kv_heads = 1` 时，GQA 退化成 MQA

## 4. Shape 和实现要点
- Q 的 head 数可以多于 K/V 的 head 数
- 需要满足 `num_heads % num_kv_heads == 0`
- `num_queries_per_kv = num_heads / num_kv_heads` 表示每组 K/V 服务多少个 query head
- `repeat_interleave` 常用于把较少的 K/V 在 head 维度上“展开”到和 Q 对齐
- 这里更接近“共享后再对齐使用”，不是真正重新学出独立 K/V

## 5. 为什么 GQA 更省
- K/V 投影参数更少
- 生成时缓存的 K/V 更少
- KV cache 更小，显存和内存带宽压力更低
- 对大模型推理尤其有价值

## 6. 怎么选 MHA / GQA / MQA
- 效果优先：MHA
- 效果和效率折中：GQA
- 极致推理效率 / 极省 KV cache：MQA
- 不能说“MQA 适合简单网络”，更准确是：**MQA 适合效率和部署开销极敏感的场景**

## 7. 你前面 notebook 的定位
- 你前面真正实现过的是：SingleHead、MHA、Decoder、GQA
- 没有单独实现一个专门版的 MQA

## 8. 一句话口述模板
- GQA 就是让多个 query head 按组共享 K/V，用更少的 K/V 来逼近 MHA 的效果，从而减少推理时的 KV cache 开销。

# 五、LoRA（Low-Rank Adaptation）答案提纲

## 1. LoRA 想解决什么问题
- 大模型全量微调成本太高：显存大、训练慢、存储贵。
- LoRA 的目标是：冻结主干大权重，只训练一个低秩增量。
- 所以它属于 **参数高效微调（PEFT）**。

## 2. LoRA 的本质
- 不是重训整个线性层。
- 而是在原线性层上加一个低秩更新：
```text
W' = W + ΔW
ΔW = BA
```
- 前向可写成：
```text
xW' = xW + xBA
```

## 3. 为什么用低秩
- 完整更新矩阵太大。
- 低秩分解用更少参数逼近有效更新方向。
- `rank r` 越小，参数越省；但太小会限制表达能力。

## 4. `lora_alpha` 的意义
- 用来控制 LoRA 分支更新量的尺度。
- 常和 `alpha / r` 一起出现，让不同 rank 下更新强度可控。

## 5. 初始化为什么常让 `ΔW = 0`
- 一开始不破坏原模型行为，更稳定。
- 通常一个矩阵随机初始化，另一个先置零，使初始增量为 0。
- 训练再逐渐学出有效增量。

## 6. 训练时谁在更新
- 原始大权重 `W` 一般冻结。
- 真正训练的是 LoRA 的小矩阵 `A/B`。
- 所以可训练参数量大幅下降。

## 7. merge / unmerge 的理解
- merge：把 `ΔW` 直接加到原始权重里，推理时就能像普通线性层一样跑
- unmerge：再把增量从主权重里拆出去
- 理论上 merge 前后数学结果应一致
- 训练时常不 merge，推理时常 merge 以简化前向和提高效率

## 8. notebook 里的实现重点
- `merge=False`：前向时走“主干 + LoRA 分支”
- `merge=True`：前向时直接用合并后的权重
- 严格等价性验证最好在 `eval()` 且关闭 dropout 时做
- 只比较输出 shape 不能证明 merge/unmerge 等价，必须比较数值结果

## 9. LoRA 可以插在哪
- attention 的 Q/K/V/O 上可以插
- FFN 也可以插
- 实际插哪里取决于任务和工程取舍

## 10. LoRA 对比全量微调
- 优势：省显存、省参数、省存储、切换任务方便
- 代价：效果未必总能完全追平全量微调
- 产物：往往不是一个全新完整模型，而是一组增量参数

## 11. 一句话口述模板
- LoRA 的本质是冻结原模型，只学习一个低秩权重增量，用很少的参数完成对大模型的适配。

# 六、跨文件综合回顾答案提纲

## 1. 五个文件的主线
- SingleHead：先理解最基本的注意力机制
- MultiHead：把单头扩展成多头并行，增强表达能力
- Decoder：把 attention 放进完整 block，理解 LLM 主干结构
- GQA：在 attention 结构上做推理效率优化
- LoRA：不是改模型结构，而是改微调方式

## 2. 三类知识要分层
- 模型结构本身：SingleHead / MultiHead / Decoder
- 工程优化：GQA
- 参数高效微调：LoRA

## 3. 作用层面怎么区分
- attention：作用在前向计算与上下文交互层面
- GQA：作用在 attention 的 K/V 设计与推理缓存层面
- LoRA：作用在参数更新方式层面

## 4. 几组高频对比
- SingleHead vs MultiHead：一个头 vs 多个头并行
- MultiHead vs GQA：独立 K/V vs 分组共享 K/V
- GQA vs MQA：分组共享 vs 全共享
- GQA vs LoRA：一个是结构/推理优化，一个是微调优化
- Decoder block vs 单独 attention：block = attention + FFN + 残差 + 归一化

## 5. Shape 能力的最低要求
- 能解释 `[B,T,C] -> [B,H,T,D] -> [B,H,T,T] -> [B,T,C]`
- 能解释 `reshape / transpose / concat / output_proj`
- 能知道 block 里为什么前后 shape 基本保持一致

## 6. 工程思维答案骨架
- 影响表达能力：MHA、FFN、整体 decoder 深度
- 影响推理效率：GQA、MQA、KV cache 设计
- 影响可训练参数量：LoRA
- 影响 KV cache 大小：MHA / GQA / MQA 的 K/V 设计

## 7. 三种目标下怎么选
- 效果优先：MHA
- 效率优先：GQA 或 MQA
- 存储 / 微调成本优先：LoRA

# 七、最终自测题的标准答案骨架

## 1. Self-Attention 的本质
- 相关性打分 + 权重归一化 + 加权聚合

## 2. 为什么要有 Q/K/V
- Q 负责提问，K 负责匹配，V 负责提供内容

## 3. Multi-Head 的价值
- 多个子空间并行建模，学习不同关系模式

## 4. Decoder 为什么还需要 FFN、残差、LayerNorm
- attention 只做 token 间交互；FFN 做 token 内变换；残差和 LN 保证深层训练稳定

## 5. causal mask 为什么关键
- 防止当前 token 偷看未来，保证训练和生成目标一致

## 6. GQA 相对 MHA 改了什么
- 把每个 head 独立 K/V 改成分组共享 K/V

## 7. GQA 为什么能减少 KV cache
- 因为缓存的 K/V 头数减少了

## 8. GQA / MQA / MHA 怎么选
- MHA：最强但最贵
- GQA：折中
- MQA：最省但更激进

## 9. LoRA 的本质
- 冻结主干，只训练低秩增量参数

## 10. merge / unmerge 为什么理论等价
- 因为本质都是在实现同一个 `W + ΔW`

## 11. 五个 notebook 的完整故事
- 从最基础 attention，到多头，到 decoder 主干，再到推理优化 GQA，最后到微调优化 LoRA

## 12. 四条线怎么一起讲
- 原理：它想解决什么问题
- shape：张量怎么变
- 代码：forward 怎么写
- 工程动机：为什么要从前一个版本演进到后一个版本
